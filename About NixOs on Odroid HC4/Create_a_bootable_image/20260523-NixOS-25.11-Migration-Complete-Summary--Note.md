# NixOS 25.11 Odroid HC4 Migration â€” Complete Summary

## Problem

NixOS SD image built for Odroid HC4 fails to boot â€” blue LED stays solid, no display output, no network response. This has been an ongoing debugging challenge spanning multiple boot strategies.

## Root Cause Discovery

### The Boot Chain Architecture

The Odroid HC4 uses the **Amlogic G12B (S905X2)** SoC, which has a strict ROM bootloader requirement. Understanding this chain is critical:

```
1. ROM bootloader (in SoC silicon)
   â”‚
   â”śâ”€â”€ Must load binary from sector 1 of SD card
   â”śâ”€â”€ Only accepts: Amlogic FIP format (magic: c8 e6 c3 d7)
   â”śâ”€â”€ Rejects: Mainline U-Boot (magic: ed 77 11 68)
   â””â”€â”€ LED behavior: solid blue = FIP accepted, no light = rejected
   â”‚
2. Hardkernel FIP (sector 1, 872,304 bytes)
   â”‚
   â”śâ”€â”€ Contains: BL2 + encrypted BL30/BL31 + U-Boot + DDR firmware
   â”śâ”€â”€ Filesystem support: FAT32 ONLY
   â””â”€â”€ Searches FAT32 root for: boot.scr (NOT /boot/boot.scr)
   â”‚
3. boot.scr (U-Boot script)
   â”‚
   â”śâ”€â”€ Must use: fatload (not load) for FAT32
   â”śâ”€â”€ Must reference: /Image, /initrd, /dtb/... (FAT32 root paths)
   â””â”€â”€ Memory addresses: must not conflict with FIP memory region
   â”‚
4. kernel + initrd + DTB (loaded by boot.scr)
   â”‚
   â””â”€â”€ kernelParams: root=LABEL=NIXOS_SD (must match fs label)
```

### Filesystem Support

**The Hardkernel FIP only supports FAT32.** This is the critical constraint that determines the entire boot strategy. Mainline U-Boot supports ext4, but mainline U-Boot is rejected by the ROM bootloader (no blue light at all).

## What We Tried

### Attempt 1: Mainline U-Boot
- Copied Armbian's `u-boot.bin` (`ed 77 11 68`) to SD card
- **Result**: No blue light â€” ROM bootloader rejected it
- **Lesson**: Can't use raw U-Boot.bin; need proper Amlogic FIP format

### Attempt 2: Hardkernel FIP with ext4 boot files
- Used Hardkernel FIP (`c8 e6 c3 d7`) at sector 1
- Placed kernel/initrd/dtb on ext4 partition 2 (`/boot/`)
- boot.scr: `load mmc 0:2 ...`
- **Result**: Solid blue â€” U-Boot loaded but can't read ext4
- **Lesson**: Hardkernel U-Boot can only read FAT32

### Attempt 3: Hardkernel FIP with FAT32 boot files, /boot/ subdirectory
- Used Hardkernel FIP at sector 1
- Placed files on FAT32 partition 1
- Tried `/boot/` subdirectory: `boot.scr` at `/boot/boot.scr`
- **Result**: Solid blue â€” U-Boot running but can't find `boot.scr`
- **Lesson**: Hardkernel U-Boot expects `boot.scr` at **root of FAT32**

### Attempt 4: Hardkernel FIP with files at FAT32 root (working concept)
- Used Hardkernel FIP at sector 1
- **Everything at FAT32 root**: `boot.scr`, `Image`, `initrd`, `dtb/`, `u-boot.bin`
- boot.scr loads from `mmc 0:1` with paths like `/Image`, `/initrd`, `/dtb/...`
- **Status**: Built but needed to test on hardware

### Attempt 5: Armbian approach analysis (different U-Boot strategy)
- Analyzed working Armbian image (`20260522-Armbian-working.img.zst`)
- **Armbian uses**: its own signed FIP (magic `f0 f1 2e ef`) built from mainline U-Boot with ext4 support
- **Armbian partition layout**: single ext4 partition, no FAT32
- **NixOS constraint**: `ubootOdroidC4` doesn't exist in nixpkgs, can't replicate Armbian approach
- **Lesson**: Must use Hardkernel blob + FAT32 since nixpkgs doesn't have Odroid C4 U-Boot

### Attempt 6: NixOS 25.11 Migration (CURRENT â€” builds successfully)
- **U-Boot**: Hardkernel blob from `blob/u-boot-odroidc4-189.tar.gz`
- **Partition layout**: FAT32 (256 MB) + ext4 (rest of disk)
- **Boot files**: FAT32 ROOT level (`/Image`, `/initrd`, `/boot.scr`, `/dtb/...`)
- **Boot script**: `fatload` commands, addresses `0x34000000`, `0x04080000`, `0x32000000`
- **Build**: Queued all 8 derivations successfully
- **Status**: Configuration evaluates correctly, ready for hardware testing

## Final Image Structure

### Partition Layout
```
FAT32   sector 2048+      (256 MB) â€” root contains: boot.scr, Image, initrd, dtb/, u-boot.bin
ext4    sector 526336+    (2.7 GB) â€” NixOS root filesystem, label "NIXOS_SD"
```

### Boot Chain
1. **ROM bootloader** â†’ reads sector 1 â†’ loads Hardkernel FIP (`c8 e6 c3 d7`)
2. **Hardkernel U-Boot** â†’ loads `boot.scr` from FAT32 root (`mmc 0:1`)
3. **boot.scr** â†’ loads kernel/initrd/dtb via `fatload` from FAT32 root (`mmc 0:1`)
4. **booti** â†’ boots kernel, which mounts root from ext4 partition 2 (`mmc 0:2`)

### boot.cmd Contents
```bash
echo ============================================
echo NixOS Odroid HC4 boot (boot.scr)
echo ============================================
fatinfo mmc 0:1
echo --------------------------------------------
setenv bootargs "console=ttyS0,115200n8 console=tty0 root=LABEL=NIXOS_SD rw rootwait rootfstype=ext4"
echo Loading kernel...

echo Loading device tree...
fatload mmc 0:1 0x04080000 /dtb/meson-sm1-odroid-hc4.dtb
echo Loading initrd...
fatload mmc 0:1 0x32000000 /initrd
setenv ramdisk_size ${filesize}
echo ============================================
echo Booting at kernel=0x34000000 initrd=0x32000000 fdt=0x04080000
booti 0x34000000 0x32000000:${filesize} 0x04080000
```

## Known Issues / Debugging Notes

### Memory Addresses
The addresses `0x08080000`, `0x01000000`, `0x13000000` were too low and likely collided with the Hardkernel FIP memory region. The corrected addresses `0x34000000`, `0x04080000`, `0x32000000` match the working Armbian configuration.

### Hardkernel FIP Limitations
- Only supports FAT32 â€” no ext4
- Must have `boot.scr` at FAT32 partition root (not in subdirectory)
- Magic bytes `c8 e6 c3 d7` must be at sector 1
- Mainline U-Boot (`ed 77 11 68`) is rejected by ROM bootloader

### Why Not Armbian Approach?
Armbian builds its own signed FIP from mainline U-Boot source with ext4 support enabled. This requires `ubootOdroidC4` in nixpkgs, which doesn't exist. We must use the Hardkernel blob + FAT32 strategy.
